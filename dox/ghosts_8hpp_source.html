21 #ifndef _GHOSTS_H
22 #define _GHOSTS_H
23 /** \file
24  * Ghost particles and particle exchange.
25  *
26  * In this file you find everything concerning the exchange of
27  * particle data (particles, ghosts, positions and forces) for short
28  * range interactions between the spacial domains of neighbouring
29  * nodes.
30  *
31  * <h2> How does this work </h2>
32  * The ghost communication transfers data from cells on one node to cells on
33  * another node during the integration process. Note that data can only be
34  * transferred from one cell to one other, and the contents of the other cell
35  * will be overwritten.
36  *
37  * The particle data that has to be transferred, and especially from where to
38  * where, heavily depends on the cell system. In ESPResSo, this is abstracted
39  * in form of ghost communicators (\ref GhostCommunicator) and ghost
40  * communications (\ref GhostCommunication). The ghost communicators represent
41  * the four communications above and consist of the data to transfer (which is
42  * determined by their type) and a list of ghost communications. The data
43  * types are described by the particle data classes:
44  * - @ref GHOSTTRANS_PROPRTS transfers the @ref ParticleProperties
45  * - @ref GHOSTTRANS_POSITION transfers the @ref ParticlePosition
46  * - @ref GHOSTTRANS_MOMENTUM transfers the @ref ParticleMomentum
47  * - @ref GHOSTTRANS_FORCE transfers the @ref ParticleForce
48  * - @ref GHOSTTRANS_RATTLE transfers the @ref ParticleRattle
49  * - @ref GHOSTTRANS_PARTNUM transfers the cell sizes
50  *
51  * Each ghost communication describes a single communication of the local with
52  * another node (or all other nodes). The data transferred can be any number
53  * of cells, there are five communication types:
54  * - @ref GHOST_SEND sends data to one other node
55  * - @ref GHOST_RECV recvs data from one other node. In the case of
56  * forces, they are added up, everything else is overwritten
57  * - @ref GHOST_BCST sends data to all nodes
58  * - @ref GHOST_RDCE recvs data from all nodes. In the case of
59  * forces, they are added up, everything else is overwritten
60  * - @ref GHOST_LOCL transfer data from a local cell to another local cell.
61  * In this case, the first half of the cells are the sending cells, the
62  * other half are the corresponding receivers.
63  *
64  * Note that for the first four communications you have to make sure that
65  * when one node sends to another, that the sender has @ref GHOST_SEND
66  * and the receiver @ref GHOST_RECV. In the case of @ref GHOST_BCST resp.
67  * @ref GHOST_RDCE, all nodes have to have the same communication type
68  * and the same master sender/receiver (just like the MPI commands).
69  *
70  * A special topic are @ref GHOST_PREFETCH and @ref GHOST_PSTSTORE. For
71  * example, if all nodes broadcast to the other, the naive implementation will
72  * be that @c n_nodes times a @ref GHOST_BCST is done with different master
73  * nodes. But this means that each time <tt>n_nodes - 1</tt> nodes wait for
74  * the master to construct its send buffer. Therefore there is the prefetch
75  * flag which can be set on a pair of recv/send operations. If the ghost
76  * communication reaches a recv operation with prefetch, the next send
77  * operation (which must have the prefetch set!!) is searched and the send
78  * buffer already created. When sending, this precreated send buffer is used.
79  * In the scenario above, all nodes create the send buffers simultaneously in
80  * the first communication step, thereby reducing the latency a little bit.
81  * The pststore is similar and postpones the write back of received data
82  * until a send operation (with a precreated send buffer) is finished.
83  *
84  * The ghost communicators are created by the cell
85  * systems.
86  */
